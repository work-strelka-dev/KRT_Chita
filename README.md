# КРТ Чита — гексагональная модель оценки территории

Инструмент рассчитывает пространственные метрики КРТ на гексагональной сетке Читы, переводит их в баллы, агрегирует по критериям и формирует:

- профильный GeoPackage для оценки **города** или **инвестора**;
- интерактивную HTML-карту с вкладками профилей;
- виртуальный общий интегральный слой, который рассчитывается в браузере из двух профильных интегралов.

Основной сценарий запуска — `python run_pipeline.py` из корня проекта.

**Актуальность описания:** 17 июля 2026 года.

## 1. Что нужно знать до первого запуска

1. **Проект сейчас ориентирован на Windows.** В Excel и коде много путей с обратными слешами. На Linux/macOS такие пути потребуют нормализации.
2. **Оба профиля должны считаться на одной и той же сетке.** Граница, размер гекса и водная маска должны быть одинаковыми. Иначе общий слой может сопоставить разные ячейки по одинаковым `hex_id`.
3. **`criteria = ...` предназначен прежде всего для отладки.** Частичный прогон перезаписывает профильный GPKG и соответствующую вкладку HTML только рассчитанными критериями.
4. **Ошибки отдельных метрик не останавливают весь запуск.** Скрипт может сохранить неполный результат. После прогона обязательно проверять итоговые `OK / SKIP / ERR`.
5. **Нормировка выполняется на нескольких уровнях:** метрика → критерий → профильный интеграл → общий интеграл. Это осознанное текущее поведение, но из-за него баллы являются относительными внутри текущей территории и текущего набора ячеек.
6. **Веса в HTML не сохраняются.** После перезагрузки страницы веса критериев и профилей снова равны 5.
7. **Внешние сервисы настраиваются один раз в локальном `.env`.** `MAPBOX_TOKEN` необязателен: без него карта использует OpenStreetMap. Для критериев с изохронами необходимо задать `VALHALLA_URL`. Скопируйте `.env.example` в `.env` и заполните значения; файл автоматически загружается и исключён из Git.

На переданном наборе файлов выполнено:

- визуальная проверка готового HTML.

Причина: в репозитории нет локального каталога `input_data/`, водной маски и готовых результатов. Поэтому README описывает фактическую реализацию кода, но не является подтверждением, что каждый внешний слой сейчас доступен и совместим по схеме.

## 4. Быстрый запуск

### 4.1. Подготовить данные

Минимально обязательны:

```text
01_vector data/bound_chita.gpkg
input_data/natural_water.gpkg
Критерии.xlsx
```

Для конкретного профиля должны присутствовать все слои, указанные в колонке `layer` соответствующего листа Excel.

### 4.2. Настроить `run_config.txt`

Полный расчёт города на гексе 200 × 200 м:

```ini
criteria_sheet = город
criteria =

hex_h_spacing = 200
hex_v_spacing = 200
```

Тест только критериев 1, 5 и 6 инвестора:

```ini
criteria_sheet = инвестор
criteria = 1, 5, 6

hex_h_spacing = 200
hex_v_spacing = 200
```

Поддерживаемые алиасы листов:

- `город`, `city`, `7.2`;
- `инвестор`, `investor`, `7.1`;
- либо полное имя листа.

### 4.3. Запустить оба профиля

```powershell
# 1. В run_config.txt поставить criteria_sheet = город
python run_pipeline.py

# 2. В run_config.txt поставить criteria_sheet = инвестор
python run_pipeline.py
```

После второго полного прогона:

```text
results/krt_scores_city.gpkg
results/krt_scores_investor.gpkg
results/krt_map.html
```

Открыть карту:

```powershell
start results\krt_map.html
```

Вкладка **«Общий»** появится только тогда, когда в HTML присутствуют и `city`, и `investor`.

## 5. Как проходит расчёт

Канонический поток в `run_pipeline.py`:

```text
run_config.txt
      ↓
выбор листа Критерии.xlsx
      ↓
построение flat-top гекс-сетки по bound_chita.gpkg
      ↓
удаление гексов, центр которых попал в natural_water.gpkg
      ↓
расчёт сырых метрик
      ↓
перевод метрик в баллы
      ↓
агрегация метрик в crit_<ID>_score
      ↓
повторная min–max-нормировка критерия
      ↓
results/krt_scores_<profile>.gpkg
      ↓
обновление профиля в results/krt_map.html
```

### 5.1. Построение сетки

- CRS расчёта жёстко задана как `EPSG:32649` — UTM zone 49N.
- Сетка flat-top.
- `hex_h_spacing` и `hex_v_spacing` — расстояния flat-to-flat в метрах.
- При равных значениях получается правильный шестиугольник.
- Гексы **не обрезаются** по границе города. Остаются полные ячейки, которые пересекают границу.
- После построения `hex_id` перенумеровывается с нуля.

### 5.2. Водная маска

Ячейка удаляется, если **центроид гекса** пересекает полигон из `input_data/natural_water.gpkg`.

Это сделано специально: береговая ячейка не удаляется из-за минимального пересечения с водой.

## 6. Контракт `Критерии.xlsx`

Используемые профильные листы:

- `7.2 Критерии (город)`;
- `7.1 Критерии (инвестор)`.

Парсер ищет строку, где первый столбец равен `№`, и начинает читать данные ниже неё. Человекочитаемые столбцы можно оформлять свободнее, но машинный блок N–U должен сохранять позиции.

| Excel | Машинное поле | Назначение |
|---|---|---|
| N | `layer` | Путь относительно `input_data` либо абсолютный путь |
| O | `spatial_method` | Способ пространственной привязки |
| P | `spatial_params` | JSON-параметры, например `{"buffer_m": 100}` |
| Q | `math_op` | Редукция/операция |
| R | `source_field_1` | Основное числовое или категориальное поле |
| S | `source_field_2` | Второе поле для `diff` |
| T | `attr_filter` | Условие pandas `query` |
| U | `out_name` | Стабильное имя сырой метрики, например `m_5_1` |

Дополнительно читаются:

- D — направление критерия;
- E — номер метрики (`pp`);
- I — направление метрики;
- J — тип шкалы;
- K — текст градации;
- L — балл.

### 6.1. Пути к данным

Относительный путь автоматически превращается в:

```text
input_data/<значение из Excel>
```

Абсолютные Windows-пути, UNC-пути и абсолютные POSIX-пути сохраняются без префикса.

`layer` может указывать:

1. на один файл;
2. на GeoPackage/FileGDB с несколькими слоями;
3. на папку.

Если указана папка, движок рекурсивно читает все поддерживаемые векторные источники, приводит их к `EPSG:32649` и объединяет перед расчётом.

Поддерживаемые расширения:

```text
.shp .gpkg .geojson .json .fgb .sqlite .kml .gml .tab .gdb
```

> [!WARNING]
> Для папки-источника ошибка чтения хотя бы одного найденного слоя приводит к ошибке всей метрики. Это намеренно строгое поведение, чтобы не получить «успешный» неполный источник без предупреждения.

### 6.2. Атрибутивные фильтры

Примеры:

```text
flag_res_avaria_snos == 1
house_condition == Аварийный
_mean < 12
```

Код автоматически заключает простые нечисловые значения справа от оператора в кавычки. Сложные выражения лучше сразу писать в синтаксисе `pandas.DataFrame.query`.

### 6.3. Предзаданные градации

После строки метрики могут идти дочерние строки без `pp` и без `out_name`.

Распознаются:

- изохроны: `До 5 минут пешком`, `От 5 до 10 минут`, `От 15 минут`;
- числа: `до 3`, `от 3 до 8`, `от 8`, единицы `м`, `км`, `°`, `%`;
- всё остальное считается категориальным значением.

Числовые интервалы полуоткрытые:

```text
до 3       → x < 3
от 3 до 8  → 3 ≤ x < 8
от 8       → x ≥ 8
```

## 7. Пространственные методы

| `spatial_method` | Реальное поведение | Типичное применение |
|---|---|---|
| `within` | У исходной геометрии берётся `representative_point`, затем точка должна лежать внутри гекса. Одна исходная фича попадает максимум в одну ячейку. | Экстенсивные показатели зданий, POI, школы |
| `intersect` | **Алиас `within`**, а не настоящее пересечение. Сохранён для совместимости со старым Excel. | Старые строки конфига |
| `intersects` | Настоящее геометрическое пересечение исходного объекта с гексом. | Площадные ограничения, ЗОУИТ |
| `buffer` | Гекс расширяется буфером `buffer_m`, затем ищутся пересечения. Одна фича может попасть в несколько ячеек. | УДС, маршруты, резерв мощности |
| `centroid_distance` | Евклидово расстояние от центроида гекса до ближайшей геометрии, метры. | Магистральные сети, неблагоприятные объекты |
| `raster_zonal` | Зональная статистика пикселей внутри гекса. | Рельеф |
| `isochrone` | Фактически обрабатывается отдельной веткой через `compute_isochrone_metric`; значения `spatial_method` в Excel для таких строк не определяют assigner. | Пешеходная/транспортная доступность |

Критическое различие:

```text
intersect  = representative_point → один гекс
intersects = реальное наложение → несколько гексов возможно
```

| `raster_median` | Медиана |
| `raster_sum` | Сумма |
| `raster_std` | Стандартное отклонение |
| `raster_range` | `max - min` |

| Операция | Ячейка без объектов/данных |

| растры | `NaN` |
| category predefined | `0` |
| изохроны | балл по умолчанию из внешней градации, иначе `0` |

## 9. Шкалы, направление и нормировка

### 9.1. Относительная метрика

Для валидных значений по всем гексам текущего запуска:

```text
direct:  score = 10 × (x - min) / (max - min)
inverse: score = 10 × (max - x) / (max - min)
```

Смысл:

- `direct`: больше сырое значение → выше балл;
- `inverse`: больше сырое значение → ниже балл.

Если все валидные значения одинаковы, все ячейки получают `0`, а в лог выводится предупреждение.

### 9.2. Предзаданная метрика

Баллы берутся из колонки L напрямую:

- изохронные пояса;
- числовые интервалы;
- категории.

На уровне самой метрики дополнительного min–max нет.

### 9.3. Балл критерия

В основном сценарии `run_pipeline.py`:

1. направление применяется **один раз** внутри каждой метрики;
2. баллы метрик усредняются с учётом доступных `NaN`;
3. веса сейчас равные, если `CriterionSpec.weights` не задан программно;
4. агрегат критерия ещё раз нормируется `direct` в диапазон 0–10.

```text
criterion_raw = Σ(metric_score_i × weight_i) / Σ(weight_i)
criterion_score = minmax(criterion_raw, direct)
```

Поэтому нельзя повторно инвертировать критерий после того, как инверсия уже применена к метрикам. В коде это специально предотвращено.

> [!IMPORTANT]
> Даже предзаданные баллы могут изменить относительный шаг после агрегации и нормировки критерия. Например, если на территории фактически встречаются только баллы 5 и 10, после min–max критерия они могут стать 0 и 10.

### 9.4. Почему `score_criterion()` и `run_pipeline.py` могут дать неидентичный финал

`score_criterion()` — библиотечный helper внутри `crt_pipeline.py`. Он собирает критерий, но не повторяет всю диагностическую и финальную логику `run_pipeline.py`.

Для производственного результата использовать:

```powershell
python run_pipeline.py
```

Не смешивать результаты двух путей без явной сверки.

## 10. Специальные ветки расчёта

### 10.1. Изохроны и Valhalla

Алгоритм:

1. точки приводятся к WGS84;
2. одинаковые координаты дедуплицируются;
3. запросы идут чанками;
4. если сервер не принимает несколько точек или возвращает меньше геометрий, весь чанк пересчитывается по одной точке;
5. временные сетевые ошибки повторяются с backoff;
6. изохроны каждого порога объединяются;
7. проверяется попадание **центроида гекса** внутрь объединённой изохроны;
8. пороги обрабатываются от большего к меньшему, поэтому меньшая зона может перезаписать балл большей.

Режимы:

```text
pedestrian / walk / foot
bicycle / bike
auto / car
multimodal
```

URL:

```text
VALHALLA_URL environment variable
        ↓
встроенный URL по умолчанию
```

Пустой ответ отдельной точки не останавливает остальные точки.

### 10.2. Взвешенное наложение ЗОУИТ

`weighted_overlap_sum` используется как родительская метрика. Под ней в Excel идут дочерние строки:

- название типа ограничения;
- один или несколько путей к слоям;
- фиксированный балл.

Правила:

- один тип ограничения даёт балл гексу только один раз;
- несколько файлов одного типа не удваивают балл;
- разные типы суммируются;
- используется настоящее `intersects`;
- сырая сумма затем проходит relative-нормировку по направлению метрики.

Один тип может ссылаться на несколько файлов:

```text
"path_1.gpkg"; "path_2.gpkg"
```

### 10.3. Приаэродромные территории

Для метрики инвестора `6.2` третья и четвёртая подзоны имеют hardcode:

```text
_mean < 12
```

То есть балл начисляется только участкам, где допустимая высота застройки ниже 12 м. Поле `_mean` обязательно.

### 10.4. Рельеф

Для `raster_relief_slope`:

```text
delta_h = raster_max - raster_min
slope_deg = degrees(atan2(delta_h, run_m))
```

По умолчанию `run_m` — эквивалентный диаметр конкретного гекса. Можно задать фиксированное значение:

```json
{"run_m": 200, "band": 1, "all_touched": false}
```

### 10.5. Категориальная метрика

Для каждой ячейки:

1. объекты привязываются к гексу;
2. значения категории переводятся в баллы;
3. выбирается наиболее частый балл;
4. при ничьей берётся максимальный балл;
5. без объектов — `0`.

### 11.1. Раздельные результаты

| Лист Excel | Ключ | Выход |
|---|---|---|
| `7.2 Критерии (город)` | `city` | `results/krt_scores_city.gpkg` |

`generate_map.py` читает существующий `results/krt_map.html`, заменяет только текущий профиль и сохраняет остальные.

### 11.2. Защита от устаревших критериев

При сборке карты список `crit_*_score` сверяется с текущим листом Excel:

- лишние колонки старого GPKG не включаются в HTML;
- отсутствующие колонки выводятся как предупреждение;
- старые плашки профиля удаляются;
- порядок критериев восстанавливается по Excel.

Это закрывает типовой случай после перенумерации: например, в городе было 14 критериев, стало 13, а `crit_14_score` остался в старом HTML.

Однако генератор не может пересчитать значения. После изменения состава или нумерации Excel всё равно нужен новый полный прогон профиля.

### 11.3. Профильный интегральный слой

В браузере:

```text
profile_raw = Σ(criterion_score × UI_weight) / Σ(UI_weight)
profile_integral = minmax(profile_raw, 0–10)
```

- начальный вес каждого критерия — 5;
- `NaN`-критерии исключаются из знаменателя конкретной ячейки;
- результат нормируется относительно всех гексов профиля.

### 11.4. Общий слой

Появляется только при наличии города и инвестора:

```text
combined_raw =
    (city_integral × city_weight + investor_integral × investor_weight)
    / (city_weight + investor_weight)

combined_integral = minmax(combined_raw, 0–10)
```

Начальные веса города и инвестора — 5 и 5.

Повторная нормировка общего результата нужна для контраста: минимум территории становится 0, максимум — 10.

### 11.5. Сопоставление профилей

Сначала используется `hex_id`, затем `feature.id`; индекс — только запасной вариант.

**Обязательное правило:** город и инвестор должны быть рассчитаны:

- по одной границе;
- с одинаковым размером сетки;
- с одинаковой водной маской;
- после одинаковой логики фильтрации гексов.

Иначе одинаковый `hex_id` может означать разные геометрии.

### 11.6. Технические детали карты

- Leaflet 1.9.4 при первом запуске скачивается и кешируется в `.leaflet_cache/`.
- При отсутствии сети и кеша HTML использует CDN-ссылки.
- Для сетки больше 5000 ячеек геометрия упрощается на 40 м перед сериализацией.
- Площадки КРТ рисуются поверх гексов чёрной обводкой без заливки.
- Файл площадок необязателен.
- Локальные шрифты встраиваются в HTML через base64; при их отсутствии используется системный fallback.
- Mapbox и OpenStreetMap доступны как подложки.
- Все данные профилей встроены внутрь HTML, поэтому файл может быть тяжёлым.

## 14. Как читать лог

Пример успешной относительной метрики:

```text
5.1  OK  sum/direct | raw=[0, 123.4] | score=[0.00, 10.00]
```

Пример пропуска:

```text
SKIP: source_field_1 не заполнен
```

Пример ошибки:

```text
ERR: не удалось загрузить слой — ...
```

Итог:

```text
OK: <число метрик>   SKIP: <число>   ERR: <число>
```

После полного прогона ожидается:

- все необходимые метрики — `OK`;
- `SKIP = 0`, кроме сознательно незаполненных строк;
- `ERR = 0`;
- для каждого критерия есть строка `→ балл критерия ...`;
- в GPKG присутствуют все ожидаемые `crit_<ID>_score`.

> [!WARNING]
> Для `weighted_overlap_sum` ошибки отдельных дочерних путей добавляются в счётчик `ERR`, но сама метрика может всё равно завершиться `OK` по успешно прочитанным слоям. Такой результат считается частичным и требует проверки.

## 16. Безопасный порядок изменений

### Изменение только текста критерия

1. Изменить название/обоснование в Excel.

3. Проверить подпись в HTML.

4. Заполнить `spatial_params`, если нужны.

2. Поставить `Предзаданный`.

## 17. Типовые проблемы

### `FileNotFoundError` для слоя

Причины:

- запуск не из корня проекта;
- неправильный относительный путь в Excel;
- лишняя папка `Data/` внутри значения;
- различие регистра;
- путь с `\` запускается на Linux/macOS;
- отсутствует sidecar-файл shapefile (`.dbf`, `.shx`, `.prj`).

Проверка:

```powershell
python -c "from pathlib import Path; print(Path('input_data/...').exists())"
```

### У слоя нет CRS

Расчёт останавливает метрику. Назначить корректную CRS в QGIS и сохранить слой заново. Не задавать CRS «на глаз».

### `KeyError` по полю

Проверить:

- `source_field_1`;
- `source_field_2`;
- `attr_filter`;
- регистр, пробелы и кириллицу в имени;
- фактическую схему слоя.

```python
import geopandas as gpd
gdf = gpd.read_file("input_data/...")
print(gdf.columns.tolist())
```

### Все баллы равны нулю

Возможные причины:

- все сырые значения одинаковы;
- слой пуст после фильтра;
- геометрии вне охвата;
- неверная CRS;
- поле прочиталось строкой и превратилось в `NaN`;
- метрика действительно однородна.

При `min == max` код намеренно ставит 0 всем ячейкам.

### Карта показывает старый профиль

`generate_map.py` сохраняет второй профиль из существующего HTML. Если сетка или методика были радикально изменены:

```powershell
del results\krt_map.html
```

Затем заново последовательно запустить город и инвестора.

### Общий слой выглядит неверно

Проверить:

- одинаковое количество гексов;
- одинаковые `hex_id`;
- одинаковую геометрию;
- одинаковый размер сетки;
- одинаковую водную маску;
- что оба профиля пересчитаны после последнего изменения.

### Valhalla не отвечает

1. Проверить `VALHALLA_URL`.
2. Открыть доступ к серверу/VPN.
3. Увеличить timeout в коде только после проверки сети.
4. Посмотреть HTTP-код.
5. Не считать отдельные нерассчитанные точки критической ошибкой всего проекта, но зафиксировать их количество.

### HTML без подложки

- Mapbox token недействителен;
- нет сети;
- Leaflet CDN недоступен и нет `.leaflet_cache`;
- можно переключиться на OSM;
- аналитические гексы и тултипы должны работать независимо от Mapbox.

### Не хватает контраста интеграла

Профильный и общий интегралы уже проходят min–max. Если карта всё ещё однородна, причина обычно в близких критериях, большом числе одинаковых нулей, пропущенных метриках или фактически равных весах, а не в отсутствии нормировки.

## 18. Известные ограничения и технический долг

1. `TARGET_CRS = 32649` и путь к границе захардкожены под Читу.
2. `run_pipeline.py` — исполняемый скрипт с кодом верхнего уровня, без CLI и без `main()`.
3. Нет полноценного набора автотестов в текущем наборе файлов. Старый бриф упоминает `test_crit1.py`, но его в репозитории сейчас нет.
4. Mapbox-токен и адрес Valhalla задаются через переменные окружения.
7. `.gitignore` не исключает `input_data/`, `01_vector data/` и `Критерии.xlsx`. Перед push нужно решить, какие данные допустимо публиковать и укладываются ли они в ограничения GitHub.
8. Shapefile состоит из нескольких файлов; нельзя коммитить только `.shp`.
9. Веса карты живут только в памяти браузера.
10. Общий слой не проверяет геометрическое равенство сеток, только идентификаторы.
11. Partial run перезаписывает полноценный профиль.
12. `score_criterion()` и `run_pipeline.py` имеют различия в финальной свёртке.
13. Имена полей и фильтры зависят от русскоязычных атрибутов и точного регистра.
14. Нет формальной фиксации лицензий исходных данных.
15. Нет файла `LICENSE` для самого кода.

## 20. Рекомендуемый рабочий процесс коллеги

1. Не менять код и Excel одновременно без контрольного прогона.
2. Сначала тестировать один критерий через `criteria = N`.
3. Читать полный лог, а не только финальное сообщение `[OK]`.
4. После локального исправления делать полный прогон профиля.
5. После изменения сетки пересчитывать оба профиля с нуля.
6. После изменения нумерации сверять Excel, GPKG и HTML.
7. Не трактовать 0 как «нет данных»: для части операций это осмысленный ноль, для отсутствия данных используется `NaN`.
8. Не инвертировать критерий повторно.
9. Не заменять `intersects` на `intersect` — это разные алгоритмы.
10. Все методологические hardcode-комментарии сохранять рядом с реализацией.
11. Перед передачей результата сохранять лог прогона вместе с датой и версией Excel.
12. Любое изменение формулы нормировки обсуждать отдельно: оно меняет все итоговые карты, даже если сырые метрики не изменились.

## 21. Краткий глоссарий

| Термин | Значение в проекте |
|---|---|
| raw | Сырое значение метрики в гексе |
| metric score | Балл конкретной метрики |
| criterion score | Агрегированный и повторно нормированный балл критерия |
| profile integral | Интеграл всех критериев города или инвестора, считается в браузере |
| combined integral | Общий интеграл города и инвестора, считается в браузере |
| direct | Чем больше raw, тем выше score |
| inverse | Чем больше raw, тем ниже score |
| predefined | Балл задан таблицей градаций |
| relative | Балл получен min–max по текущей территории |
| `NaN` | Нет валидных данных |
| 0 | Может быть осмысленным отсутствием объектов либо результатом вырожденной нормировки |

## 23. Контактная точка при аварии

Если результат внезапно изменился, проверять в таком порядке:

```text
Критерии.xlsx
→ run_config.txt
→ наличие и схема входных слоёв
→ CRS
→ raw-диапазоны в логе
→ направление metric direct/inverse
→ criterion aggregation
→ профильный integral
→ общий integral
```

Сначала локализовать уровень, на котором появилось расхождение. Не начинать с визуальной палитры карты: HTML только отображает уже рассчитанные критерии и отдельно считает интегралы.
